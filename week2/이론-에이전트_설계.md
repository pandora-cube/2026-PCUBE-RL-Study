# 이론: 에이전트 설계

## 좋은 결과를 얻기 위한 에이전트 설계

> ML-Agents 개요(`week2/이론-ml_agents_개요.md`)에서 "무엇이 있는지"를 봤다면, 이 문서는 **"어떻게 설계해야 학습이 되는가"**를 다룬다. PPO/SAC 같은 알고리즘은 우리가 짜지 않는다 — 우리가 정하는 것은 **관측·행동·보상·에피소드**, 그리고 하이퍼파라미터다. 결과의 대부분이 여기서 갈린다.
>
> 공식 문서: <https://docs.unity3d.com/Packages/com.unity.ml-agents@4.0/manual/Learning-Environments-Agents.html>

---

## 1. 왜 알고리즘보다 설계인가

- PPO는 **우리가 준 관측·행동·보상 안에서만** 최적화한다. 보상이 잘못됐으면 "잘못된 목표"에 완벽히 도달한다 → **GIGO(Garbage In, Garbage Out)**.
- 학습이 안 될 때 사람들은 하이퍼파라미터부터 만지지만, 실전에서 결과를 뒤집는 것은 대개 **문제 정의(환경 설계)** 다. 의심 순서: **보상 → 관측 → 행동 → 에피소드 → 하이퍼파라미터**.
- 아래 네 가지를 관통하는 원칙: **에이전트가 정보를 충분히 받는가(관측), 목표를 향할 수단이 있는가(행동), 목표가 보상에 정확히 반영됐는가(보상), 시행이 잘 끊기고 다양하게 리셋되는가(에피소드).**

---

## 2. 관측 설계 — 무엇을 보여주고 무엇을 가릴 것인가

가장 흔한 실패 원인: **결정에 필요한 정보를 안 줬거나, 쓸데없는 걸 잔뜩 줬다.**

**넣어야 할 것**
- 에이전트가 **최적 결정을 내리는 데 실제로 필요한 정보**. "내가 이 게임을 이 관측만 보고 플레이할 수 있나?"를 자문한다. 못 하면 관측이 부족한 것.
- **마르코프 원칙**: 현재 관측만으로 결정 가능해야 한다. 속도·방향처럼 순간값으로 안 드러나는 시간 정보가 필요하면 **관측 스태킹**(최근 N프레임)이나 LSTM으로 보강.

**빼야 할 것 / 가공할 것**
- **정규화**: 신경망 입력은 대략 `[0,1]` 또는 `[-1,1]`로 스케일. 위치 300, 속도 0.02가 섞이면 큰 값이 학습을 지배한다. (거리는 최대치로 나누고, 각도는 sin/cos로.)
- **절대좌표 대신 상대좌표**: "목표의 월드 좌표 (37, 0, 12)"보다 "목표까지의 상대 방향·거리"가 훨씬 일반화가 잘 된다. 맵이 바뀌어도 통한다.
- **중복·무관 관측 제거**: 차원이 늘면 학습이 느려지고 잡음이 낀다(차원의 저주). 결정에 안 쓰이는 값은 뺀다.
- **시각 관측은 최후의 수단**: 픽셀은 강력하지만 CNN이 무겁고 데이터를 훨씬 많이 먹는다. 수치를 알 수 있으면 **벡터**, 시야 스캔이면 **레이캐스트**를 먼저 쓴다.

> 체크: 관측을 바꿨다면 `Behavior Parameters`의 **Vector Observation Space Size**도 실제 개수와 맞춰야 한다(불일치는 흔한 버그).

## 3. 행동 설계 — 무엇을 할 수 있게 할 것인가

- **연속 vs 이산 선택**: 물리적 조작(토크·조향·힘)은 **연속**, 선택지가 분명한 조작(정지/전진/점프, 4방향)은 **이산**. 억지로 한쪽에 몰지 말 것.
- **행동 공간 최소화**: 행동이 많을수록 탐색 공간이 지수적으로 커진다. 정말 필요한 자유도만 남긴다.
- **행동 마스킹(이산)**: 지금 불가능한 선택지(벽 쪽 이동, 쿨다운 중 스킬)를 마스킹하면 탐색 낭비가 줄어 학습이 빨라진다.
- **일관성**: 같은 행동 인덱스는 상태와 무관하게 같은 의미여야 한다(브랜치0-값1이 어떤 상황에선 전진, 다른 상황에선 점프면 학습이 헷갈린다).
- **결정 주기(Decision Period)**: 매 프레임 결정할 필요는 거의 없다. 주기를 늘리면(예: 5) 한 결정이 여러 프레임을 커버해 학습이 빨라지고 행동이 안정된다. 반응성이 중요한 과제만 짧게.

---

## 4. 보상 설계 — 가장 중요하고 가장 어렵다

보상은 **목표를 수학으로 번역**하는 일이다. 여기서 대부분의 실패가 난다.

### 4-1. 희소(sparse) vs 조밀(dense/shaped)
- **희소 보상**: 목표 달성 시에만 `+1`. 정직하지만 초반에 신호가 없어 학습이 안 붙을 수 있다.
- **조밀 보상(reward shaping)**: 중간 진행에 작은 보상을 줘 방향을 알려준다(목표에 가까워지면 `+`). 학습이 빨라지지만 **reward hacking** 위험 — 에이전트가 목표 대신 보상 자체를 착취(제자리 진동으로 근접 보상 반복 수확 등). 유도는 하되 최종 목표와 어긋나지 않게.

### 4-2. 실전 규칙
- **스케일을 작게**: 개별 보상은 대략 `[-1, 1]` 근처. 매 스텝 큰 값을 누적하면 값 함수가 폭주한다.
- **시간 페널티**: 매 스텝 작은 음수(`-1/MaxStep` 정도)를 주면 "빨리 끝내라"가 학습된다.
- **성공/실패 보상은 명확하게**: 성공 `+1`, 치명적 실패 `-1`, 그리고 `EndEpisode()`.
- **여러 항의 균형**: 근접 보상·시간 페널티·성공 보상이 섞이면 서로 상쇄되지 않게 크기를 조율한다.
- **`AddReward` vs `SetReward`**: `AddReward`는 그 스텝 보상에 **누적**, `SetReward`는 그 스텝 값을 **덮어쓴다**. 스텝마다 조금씩 주며 마지막에 큰 보상을 얹는 패턴이 흔하다.

| 상황 | 보상 예시 |
| --- | --- |
| 목표 도달 | `SetReward(+1.0)` + `EndEpisode()` |
| 목표에 가까워짐 | `AddReward(+0.01 * 접근량)` (신중히 — hacking 주의) |
| 매 스텝(시간 압박) | `AddReward(-1.0 / MaxStep)` |
| 낙사·충돌 등 실패 | `SetReward(-1.0)` + `EndEpisode()` |

> **검증**: 학습 전에 `Heuristic Only`로 직접 조종하며 의도한 순간에 보상이 들어오는지 눈으로 확인한다. 가장 값싼 디버깅이다.

---

## 5. 에피소드 설계 — 시행을 어떻게 끊고 다시 시작하나

- **종료 조건 3종**: ① 성공, ② 치명적 실패, ③ **타임아웃**(`Max Step` 또는 무한 루프 방지). 종료 시 `EndEpisode()`.
- **리셋 = 일반화 기회**: `OnEpisodeBegin()`에서 에이전트·목표·장애물 위치를 **무작위화**한다. 항상 같은 배치로 시작하면 그 배치에만 과적합한다.
- **에피소드 길이**: 너무 짧으면 목표까지 도달 경험을 못 쌓고, 너무 길면 한 에피소드에 잡음이 누적돼 신용 배분이 어렵다. 과제에 맞게.
- **Max Step 설정 위치**: 에이전트별 `Max Step`(에피소드 강제 종료)과 config의 `max_steps`(전체 학습 스텝 예산)는 다른 값이다 — 혼동 주의.

## 6. 학습을 쉽게 만드는 장치

설계가 맞아도 학습이 느리거나 안 붙을 때 얹는 도구들.

| 장치 | 효과 | 언제 |
| --- | --- | --- |
| **병렬 환경(Area 복제)** | 같은 학습 영역을 씬에 여러 개 복제 → 경험 수집 N배 가속 | 거의 항상. 3DBall도 12개를 깐다. |
| **Curriculum Learning** | 난이도를 쉬운 것부터 점진 상승 | 처음부터 어려워 신호가 안 잡힐 때 |
| **Env Parameter Randomization** | 환경 변수(중력·크기·마찰 등) 무작위화 | 과적합 방지·일반화·sim-to-real |
| **Demonstration + BC/GAIL** | 사람 시연으로 초기 정책 부트스트랩 | 희소 보상·탐색이 어려운 과제 |
| **관측 스태킹 / LSTM** | 시간 정보·기억 보강 | 부분 관측(POMDP)일 때 |

> **병렬 환경 주의**: 복제된 각 영역이 서로 독립적이어야 한다(물리·트리거가 옆 영역에 새지 않게). 같은 **Behavior Name**을 공유해야 하나의 정책으로 합쳐 학습된다.

## 7. 하이퍼파라미터 — PPO 지식으로 감 잡기

설계가 끝난 뒤 마지막으로 만지는 것. 우리가 배운 PPO 개념이 그대로 노브가 된다.

| config 키 | PPO 개념 | 감 잡기 |
| --- | --- | --- |
| `epsilon` | **클리핑 범위 ε** (PPO 5장) | 보통 `0.2`. 크면 공격적·불안정, 작으면 느림·안정. |
| `beta` | **엔트로피 계수**(탐험 보너스) | 탐험 부족(엔트로피가 너무 빨리 0)→↑, 계속 무작위→↓. |
| `lambd` | **GAE λ** | 보통 `0.95`. 편향-분산 절충. |
| `gamma` | **할인율 γ** | `0.99` 기본. 장기 보상 중요하면 ↑(0.995), 근시안 과제면 ↓. |
| `num_epoch` | 경험 재사용 횟수 | PPO가 한 배치를 여러 번 쓰는 그 횟수. 보통 3. 크면 과적합/불안정. |
| `buffer_size` / `batch_size` | 업데이트 전 모을 경험량 / 미니배치 | buffer가 클수록 안정·느림. 연속 행동은 크게. |
| `learning_rate` | 학습률 | `3e-4` 근방. 발산하면 ↓. |
| `hidden_units` / `num_layers` | 신경망 크기 | 관측이 단순하면 작게(128/2), 복잡·시각이면 크게. |
| `max_steps` | 총 학습 스텝 예산 | 과제 난이도에 맞춰. 곡선이 아직 오르면 늘린다. |

> **순서**: 하이퍼파라미터는 **마지막에** 만진다. 학습이 전혀 안 붙으면 십중팔구 문제는 보상·관측이지 `learning_rate`가 아니다.

## 8. 디버깅 & 체크리스트

### TensorBoard로 보는 것 (`tensorboard --logdir results`)
- **Cumulative Reward**가 우상향하는가 → 학습의 1차 지표. 평평하면 보상/관측 의심.
- **Entropy**가 서서히 하강하는가 → 탐험→수렴. 너무 빨리 0이면 `beta`↑. 안 내려가면 신호 부족.
- **Value Loss** → 초반 상승 후 하강이 정상. 계속 폭주하면 보상 스케일 과대.
- **Episode Length** → 과제에 따라 목표 방향으로 움직이는지.

### 학습이 안 될 때 의심 순서
1. **보상**: 의도한 순간에 들어오는가? 스케일이 과하지 않나? hacking 여지는? (`Heuristic`으로 확인)
2. **관측**: 이 관측만으로 사람이 풀 수 있나? 정규화·상대좌표 됐나? Space Size 일치하나?
3. **행동**: 목표를 이룰 수단이 실제로 있나? 공간이 과하게 크지 않나?
4. **에피소드**: 리셋이 다양한가? 종료가 제때 되나? 길이가 적절한가?
5. **연결**: Behavior Name ↔ config 키 일치? (안 맞으면 아예 시작 안 됨)
6. **그다음에야** 하이퍼파라미터.

### 설계 체크리스트 (착수 전 한 번 훑기)
- [ ] 이 관측만으로 과제를 풀 수 있다(사람 기준).
- [ ] 관측이 정규화·상대좌표로 되어 있다.
- [ ] 행동 공간이 최소한이다. 불가능한 이산 행동은 마스킹한다.
- [ ] 성공/실패 보상이 명확하고 스케일이 작다. hacking 여지를 점검했다.
- [ ] 종료 조건(성공/실패/타임아웃)이 모두 있고 리셋이 무작위화된다.
- [ ] 병렬 학습 영역이 서로 독립이고 Behavior Name을 공유한다.
- [ ] `Heuristic Only`로 관측·보상을 눈으로 검증했다.
