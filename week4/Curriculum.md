# 커리큘럼 러닝 (Curriculum Learning)

강화학습에서 **쉬운 과제부터 점점 어려운 과제로 단계적으로 올려가며 학습시키는 기법**이다.
사람이 구구단을 떼고 나서 방정식을 배우듯, 에이전트도 처음부터 최종 난이도로 던지면
학습이 안 되는 문제를 완화한다.

---

## 1. 왜 필요한가

RL이 안 풀리는 흔한 원인은 **보상이 너무 드물다 sparse reward**는 것이다.
최종 과제에서 "성공" 신호를 우연히라도 만나기까지 너무 오래 걸리면, 에이전트는
무엇이 좋은 행동인지 배울 단서를 못 얻고 무작위 탐색만 반복한다.

커리큘럼은 이 문제를 **초기 단계에서 성공을 쉽게 만들어** 푼다.

- 쉬운 단계에선 성공 신호를 자주 만나 → 기초 스킬(예: 목표에 접근, 도구 조작)을 빠르게 습득.
- 그 스킬을 발판 삼아 다음 단계의 더 어려운 문제를 탐색.
- 결과적으로 처음부터 최종 난이도로 학습할 때보다 **더 빨리, 더 안정적으로, 더 높은
  최종 성능**에 도달하는 경우가 많다.

### 언제 쓰고, 언제 안 쓰나

| 커리큘럼이 잘 듣는 경우 | 굳이 필요 없는 경우 |
|---|---|
| 보상이 드물다 (성공을 거의 못 만남) | 보상이 이미 촘촘하다(dense) |
| 최종 과제가 여러 하위 스킬의 합성 | 과제가 단순/단일 스킬 |
| 난이도를 조절할 "손잡이"가 환경에 있다 | 난이도를 매끄럽게 바꿀 축이 없다 |
| 처음부터 최종 난이도면 학습이 정체됨 | 그냥 학습해도 잘 오른다 |

> 커리큘럼은 공짜가 아니다. 단계 설계·문턱값 튜닝이라는 새 하이퍼파라미터가 생긴다.
> **먼저 커리큘럼 없이 돌려보고, 정체될 때 도입**하는 게 보통 이득이다.

---

## 2. 난이도를 조절하는 "축"

커리큘럼의 핵심은 **환경의 난이도를 연속적으로 바꿀 수 있는 파라미터(손잡이)**를 하나
이상 정하는 것이다. 자주 쓰는 축:

- **목표까지 거리 / 시간 여유** — 가까운 목표 → 먼 목표, 넉넉한 제한시간 → 촉박하게.
- **상대의 강도** — 없음/고정 봇 → 약한 상대 → 동등한 상대(self-play).
- **환경의 어수선함** — 장애물 수, 노이즈, 초기 위치의 무작위성.
- **보상의 밀도** — 초반엔 촘촘한 shaping 보상 → 후반엔 성긴 최종 보상만.
- **에이전트 수 / 과제 규모** — 1개 → 여러 개, 작은 맵 → 큰 맵.

좋은 축의 조건: **(1) 난이도를 매끄럽게 조절**할 수 있고, **(2) 쉬운 끝에서 성공이 정말
쉬우며**, **(3) 어려운 끝이 진짜 풀고 싶은 최종 과제**일 것.

---

## 3. ML-Agents에서의 표현

ML-Agents는 커리큘럼을 **환경 파라미터(environment parameter)**로 구현한다.
난이도 손잡이를 float 하나로 정의하고, 학습 설정(YAML)에서 그 값을 단계별로 올린다.
환경(C#)은 그 값을 읽어 씬 난이도를 바꾼다.

### (a) 학습 설정 — `environment_parameters`

```yaml
environment_parameters:
  difficulty:                 # 내가 정의한 난이도 손잡이 이름
    curriculum:
      - name: Stage0
        completion_criteria:  # 이 단계를 "졸업"하는 조건 (아래 4장)
          measure: reward
          behavior: MyBehavior
          signal_smoothing: true
          min_lesson_length: 100
          threshold: 0.7
        value: 0.0            # 이 단계일 때 difficulty = 0.0
      - name: Stage1
        completion_criteria: { measure: reward, behavior: MyBehavior,
                               signal_smoothing: true, min_lesson_length: 100, threshold: 0.5 }
        value: 1.0
      - name: Stage2          # 마지막 단계엔 completion_criteria 없음 → 끝까지 유지
        value: 2.0
```

### (b) 환경 — 파라미터를 읽어 난이도 반영 (C#)

```csharp
// 에피소드 리셋 시점에 현재 단계 값을 읽는다. 학습기(Python)가 없으면 기본값 사용.
var difficulty = Academy.Instance.EnvironmentParameters.GetWithDefault("difficulty", 2f);
// difficulty 값에 따라 목표 거리 / 상대 수 / 장애물 등을 세팅...
```

트레이너가 단계 조건을 만족했다고 판단하면 `difficulty` 값을 다음 단계 `value`로 바꿔
보내고, 환경은 다음 리셋 때 그 값을 읽어 난이도를 올린다.

---

## 4. 다음 단계로 넘어가는 기준 ★

각 단계의 `completion_criteria` 블록이 **졸업 조건**이다.

| 필드 | 뜻 |
|---|---|
| `measure` | 무엇으로 판단하나. `reward`(누적 보상) 또는 `progress`(전체 스텝 진행률 0~1) |
| `behavior` | 어느 정책의 값을 볼지 |
| `threshold` | 그 값이 이걸 넘으면 졸업 |
| `min_lesson_length` | 최소 이만큼(에피소드)은 지나야 졸업 허용 |
| `signal_smoothing` | true면 과거값과 가중평균해 노이즈로 튀는 걸 방지 |

### 진급 = 두 조건을 **동시에** 만족

1. **`min_lesson_length` 이상** 에피소드를 채웠고, *그리고*
2. 지정한 **`measure`의 (스무딩된) 값 > `threshold`**

둘 다 만족하는 순간 다음 단계로 넘어간다. `min_lesson_length`는 "운 좋게 몇 판 잘했다고
바로 졸업"하는 걸 막는 안전장치다.

### `measure` 두 가지

- **`reward`** — 실력 기반 진급. "이 단계를 잘하게 됐으면 올린다." 대부분의 경우 이걸 쓴다.
- **`progress`** — 시간 기반 진급. "전체 학습의 몇 %가 지나면 올린다." 실력과 무관하게
  일정에 따라 난이도를 올리고 싶을 때. 최종 스텝 대비 비율(0~1)이 threshold를 넘으면 졸업.

> 마지막 단계엔 `completion_criteria`를 두지 않는다. 더 올라갈 곳이 없으니 학습이
> 끝날 때까지 그 난이도를 유지한다.

---

## 5. 모니터링 & 튜닝

- `threshold`, `min_lesson_length`, 단계 수/간격은 전부 **튜닝 대상**이다. 정답 없음.
- TensorBoard에서:
  - **`Environment/Cumulative Reward`** — 지정한 measure가 실제로 오르는지.
  - **`Environment/Lesson Number`** — 언제 진급했는지(계단 그래프). 진급 타이밍 점검용.
- 흔한 조정:
  - 진급이 **너무 빠름**(기초 못 배우고 넘어가 다음 단계에서 무너짐) → `threshold` ↑.
  - **안 넘어감**(문턱이 그 단계에서 도달 불가능) → `threshold` ↓ 또는 단계를 더 잘게.
- **단계 난이도가 오를수록 도달 가능한 보상 자체가 낮아진다.** 그래서 threshold를 모든
  단계에 똑같이 두면 어려운 단계에서 영영 못 넘어간다 — 보통 뒤 단계일수록 낮춘다.

---

## 6. 흔한 함정

- **파국적 망각(catastrophic forgetting).** 난이도를 너무 크게 점프시키면 앞 단계에서
  배운 걸 잊고 무너진다. 단계 간 난이도 차를 완만하게, 또는 이전 난이도를 가끔 섞는다.
- **너무 잘게 쪼갬.** 단계가 많을수록 튜닝할 문턱값도 늘고 관리가 어려워진다. 3~4단계로
  시작하는 게 보통 충분하다.
- **잘못된 measure 신호.** shaping 보상이 measure를 오염시키면(예: 목표와 무관한 행동에
  보상이 쌓임) 실력이 안 늘었는데 threshold를 넘겨 잘못 진급한다.
- **쉬운 끝이 안 쉬움.** 0단계에서도 성공을 거의 못 만나면 커리큘럼의 의미가 없다.
  1단계는 "거의 무조건 성공"할 만큼 쉬워야 한다.
- **최종 과제와의 불일치.** 마지막 단계가 실제 풀려는 문제와 다르면, 잘 배워도 목표
  과제에선 못 쓴다. 커리큘럼의 끝 = 진짜 과제여야 한다.
